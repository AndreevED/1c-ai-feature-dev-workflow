# 1C AI Feature Dev Workflow

Воркфлоу для разработки доработок в **существующей** конфигурации 1С:Предприятие. Анализирует кодовую базу, проектирует архитектуру, пишет код с соблюдением стандартов и проверяет качество.

## Для чего

Набор агентов помогает дорабатывать **уже существующую** конфигурацию 1С:
- Изучает существующие паттерны и архитектуру
- Предлагает варианты реализации с учётом вашей кодовой базы
- Пишет код, который соответствует вашим стандартам
- Проверяет код на баги и соответствие правилам проекта

## Команды

```bash
# Полный цикл разработки доработки
/1c-feature-dev Добавить интеграцию с API складской системы

# Проработка идей и вариантов перед реализацией
/brainstorm Хочу добавить систему уведомлений
```

## Основной воркфлоу (9 фаз)

### Фаза 0: Оценка сложности
Оценивается масштаб задачи (простая/средняя/сложная/критичная). От этого зависит количество агентов на последующих фазах.

### Фаза 1: Discovery
Уточнение задачи — что нужно построить. Если запрос неясен, задаются вопросы о проблеме, требованиях и ограничениях. Результат сохраняется в `.tasks/task-[name]/phase1-requirements.md`.

### Фаза 2: Исследование кодовой базы
**Адаптивный подход** — количество агентов `1c-code-explorer` зависит от сложности:
- **Простая задача**: 1 агент
- **Средняя**: 1-2 агента
- **Сложная**: 2-4+ агента

Возможна итеративная работа: каждый следующий агент углубляется в находки предыдущих. Агенты возвращают оркестратору список ключевых файлов для чтения и сводку паттернов. Результат сохраняется в `phase2-exploration.md`.

### Фаза 3: Уточняющие вопросы
На основе найденного выявляются неоднозначности: граничные случаи, обработка ошибок, точки интеграции, производительность. Все вопросы представляются списком, **ждём ответов пользователя перед проектированием**. Результат сохраняется в `phase3-clarifications.md`.

### Фаза 4: Архитектура
**Адаптивный подход** — количество агентов `1c-code-architect` зависит от сложности:
- **Простая задача**: 1 агент с практичным планом
- **Средняя**: 1 агент (опционально 2 с разными подходами)
- **Сложная/критичная**: 2-3 агента с разным фокусом (минимальные изменения, чистая архитектура, баланс)

План включает этапы реализации с критериями приемки. Результат сохраняется в `phase4-architecture.md`.

### Фаза 5: Ревью плана
Агент `1c-code-architect` проверяет план **ДО реализации**: полноту, корректность, реалистичность, соответствие требованиям. Если находятся проблемы — план обновляется и ревью повторяется. **Реализация начинается только после одобрения пользователя.** Результат сохраняется в `phase5-plan-review.md`.

### Фаза 6: Реализация по этапам
Агент `1c-code-writer` реализует доработку атомарными этапами:
- Каждый этап завершается за 1 сессию
- Имеет проверяемые критерии приемки
- Отмечается в плане (🔄 в процессе, ✓ выполнено)
- Если критерии не выполнены — запускается исправление

> **Важно:** Агенты пишут только код. Все изменения метаданных выполняет пользователь до начала этапа реализации.

### Фаза 7: Ревью кода
**Адаптивный подход** — количество ревьюеров зависит от размера задачи:
- **Простая**: `1c-code-reviewer` (базовая проверка)
- **Средняя/сложная**: `1c-code-architect` (соответствие плану) + `1c-code-reviewer` (качество кода)
- **Очень большая**: раздельное ревью по модулям

Проблемы представляются пользователю: исправить сейчас, позже или продолжить как есть. При исправлении — ревью повторяется. Результат сохраняется в `phase7-code-review.md`.

### Фаза 8: Итоги
Резюме в `phase8-summary.md`: что построено, ключевые решения, изменённые файлы, технический долг, следующие шаги.

## Агенты

| Агент | Назначение |
|-------|------------|
| `1c-code-explorer` | Анализ существующей кодовой базы, трассировка выполнения |
| `1c-code-architect` | Проектирование архитектуры и ревью планов |
| `1c-code-writer` | Написание кода по правилам из `1c-rules.md` |
| `1c-code-reviewer` | Проверка кода на баги и соответствие правилам |
| `1c-code-simplifier` | Упрощение кода после написания, улучшение читаемости, элегантности |

## Использование агентов отдельно

Не обязательно запускать полный воркфлоу — агентов можно использовать по отдельности:

```bash
# Только исследовать код
"Запусти 1c-code-explorer для анализа того, как работает авторизация"

# Только спроектировать
"Запусти 1c-code-architect для проектирования слоя кеширования"

# Только написать код
"Запусти 1c-code-writer для реализации процедуры XYZ"

# Только проверить код
"Запусти 1c-code-reviewer для проверки моих последних изменений"

# Только упростить код
"Запусти 1c-code-simplifier для упрощения этого модуля"
```

## Установка

Скопируйте файлы агентов, скиллов и правил в соответствующую директорию вашего инструмента.

## Кастомизация

Этот проект можно использовать как **базу для построения собственного workflow**. Все компоненты адаптируемы под ваши задачи:

### Изменение промптов агентов

Промпты каждого агента находятся в файлах `agents/*.md`. Вы можете:
- Добавить или убрать инструкции под свой стиль работы
- Изменить формат вывода результатов
- Добавить специфичные для вашей конфигурации паттерны

### Адаптация правил

Файлы правил содержат стандарты кодирования:
- `rules/1c-rules.md` — правила написания кода на 1С
- `rules/common-rules.md` — общие правила работы

Вы можете добавить свои правила, изменить форматирование, указать специфичные соглашения.

### Модификация скиллов

Основной workflow определён в `skills/1c-feature-dev/SKILL.md`:
- **Добавление фаз** — вставьте новую фазу в нужное место воркфлоу
- **Удаление фаз** — уберите ненужные этапы
- **Изменение логики** — поменяйте порядок фаз или условия переходов между ними

### Добавление новых агентов

Создайте файл `agents/my-agent.md` по аналогии с существующими:

```markdown
---
name: my-agent
description: "Краткое описание назначения агента"
model: opus
color: blue
---

Ты — эксперт в области X. Твоя задача — ...
```

Затем используйте агента в скилле через текстовые команды:

```markdown
Запусти агента `my-agent` для анализа ...
```

## Когда использовать

**Доработки в существующей конфигурации**:
- Расширение функциональности
- Интеграции с внешними системами
- Сложная бизнес-логика
- Неясные требования

**Не для простых задач**:
- Однострочные исправления
- Тривиальные изменения
- Срочные хотфиксы

## Требования

- Инструмент с поддержкой субагентов (Claude Code и др.)
- Конфигурация 1С, выгруженная в файлы
- MCP-сервер справки 1С (опционально, используется агентами `1c-code-writer`, `1c-code-reviewer` и `1c-code-simplifier` для проверки синтаксиса и т.д.)

## Особенности

- **Адаптивность** — количество агентов зависит от сложности задачи (простая = меньше агентов, сложная = больше)
- **Ревью плана ДО реализации** — агент проверяет архитектурный план перед написанием кода, экономя время на переделках
- **Артефакты каждой фазы** — все результаты сохраняются в `.tasks/task-[name]/` (требования, исследование, план, ревью, итоги)
- **Атомарные этапы реализации** — каждый этап с критериями приемки, отмечается в плане (🔄 в процессе, ✓ выполнено)
- **Итеративное ревью** — пишем → проверяем → исправляем → повторяем до достижения качества
- **Возврат к задаче** — все артефакты в файлах, можно вернуться к задаче позже
